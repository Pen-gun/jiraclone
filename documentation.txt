Jira Clone Technical Documentation

Version: March 2026

Overview
- The project uses Next.js App Router for UI and Hono mounted at `app/api/[...route]/route.ts` for API endpoints.
- Authentication is database-session based using Prisma and PostgreSQL.
- Frontend data reads and mutations are managed with TanStack React Query.
- Workspace creation UI is implemented and currently persists to the `Project` model.

Architecture

1. UI layer
- Route pages are thin wrappers.
- Feature components handle forms and user interaction.
- Shared primitives live in `components/ui`.

2. API layer
- Hono routes are grouped under `/api/auth` and `/api/workspaces`.
- Implemented endpoints:
  - `POST /api/auth/login`
  - `POST /api/auth/register`
  - `POST /api/auth/onboarding`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`
  - `POST /api/workspaces`

3. Data layer
- Prisma models include `User`, `Session`, `Project`, `Task`, and `Comment`.
- Session rows contain `userId`, `createdAt`, `updatedAt`, and `expiresAt`.
- Cookie `auth_token` stores the session id.

4. Validation strategy
- Zod schemas in `features/schemas.ts` define request contracts.
- Hono `zValidator` enforces server-side payload validation.
- React Hook Form + zodResolver enforces client-side form validation.

Authentication Flow

1. Register
- Validate payload.
- Create `User` row.
- Create `Session` row.
- Set `auth_token` cookie.

2. Login
- Validate payload.
- Verify user credentials.
- Create `Session` row.
- Set `auth_token` cookie.

3. Resolve current user
- Client path:
  - `GET /api/auth/me` uses `sessionMiddleware`.
  - Middleware reads `auth_token`, looks up session, checks expiry, returns user.
- Server path:
  - `features/action.ts` reads `auth_token` from Next cookies.
  - It looks up session directly with Prisma for page-level auth gating.

4. Logout
- `POST /api/auth/logout` uses `sessionMiddleware`.
- Deletes the session row if found.
- Clears cookie on response.

Workspace Flow

1. Create workspace
- `CreateWorkspaceForm` validates `name` and optional `description` with Zod.
- `useCreateWorkspace` calls `POST /api/workspaces`.
- Route uses `sessionMiddleware` and creates a `Project` row with `ownerId` of current user.
- On success, form resets and `workspaces` query key is invalidated.

Frontend Integration

1. API hooks
- Auth hooks live in `features/auth/api`:
  - `use-current.ts`
  - `use-login.ts`
  - `use-logout.tsx`
  - `use-onboarding.ts`
  - `use-register.ts`
- Workspace hooks live in `features/workspaces/api`:
  - `use-create-workspace.ts`

2. UI behavior
- Submit handlers trigger mutation hooks.
- Success path typically shows toast and routes to next page.
- Error path shows toast with error details.
- Buttons use pending state to prevent duplicate submissions.

Environment Variables

Required:
- `DATABASE_URL`
- `NEXT_PUBLIC_API_URL`
- `NODE_ENV`

Cookie Notes
- Current code sets `httpOnly: true`, `sameSite: "Lax"`, and `secure: true`.
- In localhost over HTTP, `secure: true` can block session cookies.
- Production should keep `secure: true`; local development should use environment-aware cookie security settings.

Not implemented

Known Risks

1. Plain text passwords
- Passwords are currently stored and compared in plain text.

2. Cookie defaults
- Cookie flags are not yet centralized by environment.
- Current `secure: true` behavior can break localhost auth persistence.

3. Hook error consistency
- Some hooks still throw non-Error objects.

Recommended Next Steps

1. Security
- Add bcrypt/argon2 password hashing.
- Use environment-based cookie policy.

2. Quality
- Add integration tests for auth routes.
- Add e2e auth flow test.

3. Product foundation
- Add project/task list, detail, update, and delete APIs and pages.
- Add issue-style status transitions and board/list views.
