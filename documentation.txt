# Jira Clone - Current Project Documentation

Last updated: 2026-03-27
Status: In-progress prototype with working authentication + onboarding + basic workspace creation

## 1) Project Overview

This project is a Jira-inspired web app built with Next.js, React, TypeScript, Hono, Prisma, and PostgreSQL.

Current main purpose:
- Authenticate users (sign up, sign in, sign out)
- Collect onboarding profile data
- Create and list workspaces for authenticated users
- Route authenticated users to their first workspace

What this means right now:
- The app has a working account and session system.
- Workspace creation exists at API + UI component level.
- Task and comment features are defined in database schema but not implemented in UI/API yet.

## 2) Tech Stack

- Framework: Next.js 16 (App Router)
- Language: TypeScript
- UI: React 19 + Tailwind CSS v4 + custom UI primitives (Radix/shadcn-style)
- API layer: Hono (mounted under Next.js route handler)
- Database: PostgreSQL
- ORM: Prisma
- Data fetching/state: TanStack React Query
- Validation/forms: Zod + React Hook Form

## 3) Setup Instructions (Reproduce Current State)

### Prerequisites

- Node.js 20+ (recommended)
- npm
- PostgreSQL running locally (or another reachable PostgreSQL instance)

### Step 1: Install dependencies

Run from project root:

```bash
npm install
```

### Step 2: Configure environment variables

Create/update two files in the project root:

1. `.env`
```env
DATABASE_URL="postgresql://<username>:<password>@<host>:<port>/<database>?schema=public"
JWT_SECRET="replace-with-a-real-secret"
```

2. `.env.local`
```env
NEXT_PUBLIC_API_URL="http://localhost:3000"
```

Notes:
- `DATABASE_URL` is required by Prisma and server-side database code.
- `NEXT_PUBLIC_API_URL` is required by the Hono RPC client in `lib/rcp.ts`.
- `JWT_SECRET` currently exists in environment config but is not used in code yet.

### Step 3: Generate Prisma client + run migrations

```bash
npx prisma generate
npx prisma migrate dev
```

### Step 4: Start development server

```bash
npm run dev
```

Open: `http://localhost:3000`

### Useful scripts

```bash
npm run dev
npm run build
npm run start
npm run lint
```

## 4) Current Code Structure

This section explains the important folders as they exist now.

### `app/`

- `app/layout.tsx`
	- Global layout.
	- Adds query provider and toast UI.

- `app/(auth)/`
	- Auth pages and layout:
	- `sign-in/page.tsx`
	- `sign-up/page.tsx`
	- `onboarding/page.tsx`
	- `layout.tsx` contains top nav and auth-page wrapper.

- `app/(dashboard)/`
	- Protected dashboard area:
	- `layout.tsx` renders sidebar + navbar.
	- `page.tsx` does server-side redirect logic based on auth/workspaces.
	- `workspaces/[workspaceId]/page.tsx` currently placeholder UI.

- `app/api/[...route]/route.ts`
	- Hono app entrypoint mounted at `/api`.
	- Routes currently include `/api/auth/*` and `/api/workspaces/*`.

### `features/`

- `features/action.ts`
	- Server helper to resolve current user from session cookie.

- `features/schemas.ts`
	- Zod schemas for sign-in, sign-up, onboarding.

- `features/auth/`
	- `server/route.ts`: auth API endpoints.
	- `api/*`: React Query hooks for auth calls.
	- `components/*`: sign-in/sign-up/onboarding/dashboard user UI.
	- `constant.ts`: auth cookie name.

- `features/workspaces/`
	- `server/route.ts`: create/list workspace endpoints.
	- `action.ts`: server helper to get user workspaces.
	- `api/*`: React Query hooks.
	- `components/*`: workspace form + avatar.
	- `hooks/use-workspace-id.ts`: route param helper.

- `features/members/tpyes.ts`
	- Defines `MemberRole` enum used for workspace membership role.
	- File name contains typo (`tpyes`).

### `lib/`

- `prismaHelper.ts`: Prisma client setup with PostgreSQL adapter.
- `rcp.ts`: typed Hono client configured with cookies (`credentials: include`).
- `session-middelware.ts`: request auth middleware for protected endpoints.
- `sesson-cleanup.ts`: session cleanup utility (currently not wired up).
- `utils.ts`: className merge helper + invite code generator.

### `components/`

- Layout/navigation: `sidebar.tsx`, `navbar.tsx`, `mobile-sidebar.tsx`, `navigation.tsx`, `workspace-switcher.tsx`
- Utility UI: `toaster.tsx`, `spinner.tsx`, `responsive-modal.tsx`, `dotted-seperator.tsx`
- `components/ui/`: reusable base UI primitives.

### `prisma/`

- `schema.prisma`: data models and enums.
- `migrations/*`: migration history.

### `generated/prisma/`

- Generated Prisma client/types from schema. Do not edit manually.

## 5) Current Functionality (What Works Today)

### A) Authentication

Implemented endpoints in `features/auth/server/route.ts`:

- `POST /api/auth/register`
	- Creates user with email/password.
	- Creates session row.
	- Sets `auth_token` cookie.

- `POST /api/auth/login`
	- Validates credentials.
	- Creates new session row.
	- Sets `auth_token` cookie.

- `POST /api/auth/onboarding` (protected)
	- Updates `fullName`, `age`, `bio`.
	- Marks `onBoardingCompleted` true.

- `POST /api/auth/logout` (protected)
	- Deletes current session.
	- Clears auth cookie.

- `GET /api/auth/me` (protected)
	- Returns current authenticated user from middleware context.

Client hooks used by forms/components:
- `useLogin`, `useRegister`, `useOnboarding`, `useLogout`, `useCurrent`

### B) Session handling

- Session model exists in DB (`Session` table).
- `sessionMiddleware` checks cookie -> session -> expiry -> user.
- Expired sessions are deleted when detected.
- Auth cookie key is `auth_token`.

### C) Workspace flow

Implemented endpoints in `features/workspaces/server/route.ts`:

- `GET /api/workspaces` (protected)
	- Returns workspaces for current user via membership table.

- `POST /api/workspaces` (protected)
	- Creates workspace.
	- Generates invite code.
	- Adds creator as `ADMIN` in `WorkspaceMember`.

Client-side:
- `CreateWorkspaceForm` submits workspace creation.
- `WorkspaceSwitcher` displays user workspaces and route-switches by id.

### D) Routing behavior on home page

In `app/(dashboard)/page.tsx`:
- If not authenticated -> redirect to `/sign-in`
- If authenticated and has 0 workspaces -> redirect to `/workspaces/create`
- If authenticated and has workspaces -> redirect to first workspace id

Important current gap:
- `/workspaces/create` page route is not implemented yet, even though redirect expects it.

## 6) Usage Examples (How to Run/Test Now)

Use these manual tests after running `npm run dev`.

### Example 1: New user flow

1. Visit `/sign-up`
2. Register with email + password
3. You should be redirected to `/onboarding`
4. Submit full profile form
5. You should be redirected to `/`

Expected result:
- If your account has no workspace, redirect attempts to `/workspaces/create`.

### Example 2: Existing user login

1. Visit `/sign-in`
2. Enter valid credentials
3. On success, app does `router.replace('/')` + `router.refresh()`
4. Home page redirects based on workspace availability

### Example 3: Logout

1. Open dashboard top-right avatar menu
2. Click log out
3. Session is deleted and auth cookie removed
4. Protected views should push back to sign-in

### Example 4: API smoke checks (optional)

You can test endpoints with any REST client:

- `POST /api/auth/register`
- `POST /api/auth/login`
- `GET /api/auth/me` (requires auth cookie)
- `POST /api/workspaces` (requires auth cookie)
- `GET /api/workspaces` (requires auth cookie)

## 7) Current Data Model (Simple View)

Main entities in `prisma/schema.prisma`:

- `User`
- `Session`
- `Workspace`
- `WorkspaceMember`
- `Project`
- `Task`
- `Comment`

Relationship highlights:
- User owns workspaces and projects.
- User joins workspaces via `WorkspaceMember` with role enum (`MEMBER`/`ADMIN`).
- Workspace has projects.
- Project has tasks.
- Task has comments.

## 8) Known Issues / Limitations (As Of This Version)

- Passwords are currently stored and compared as plain text.
- Auth cookie uses `secure: true` in auth routes, which can break localhost over HTTP.
- `/workspaces/create` redirect target exists in logic but page is missing.
- Workspace detail page is currently placeholder only.
- Navigation active state is hardcoded (`isActive = false`).
- Session cleanup utility exists but is not currently wired into app startup.
- Several filenames contain typos (`tpyes`, `middelware`, `sesson`, `seperator`).

## 10) Quick Onboarding For A New Developer

If you are new to this codebase, do this first:

1. Read sections 1, 3, and 5 in this document.
2. Run setup steps exactly in section 3.
3. Walk through the manual tests in section 6.
4. Open these files in order:
	 - `app/api/[...route]/route.ts`
	 - `features/auth/server/route.ts`
	 - `features/workspaces/server/route.ts`
	 - `features/action.ts`
	 - `app/(dashboard)/page.tsx`
5. Then move to UI components under `features/auth/components` and `components/sidebar.tsx`.

This order helps you understand backend flow first, then frontend behavior.
