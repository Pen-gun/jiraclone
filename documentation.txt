Jira Clone Technical Documentation

Version: March 2026

Overview
- The project uses Next.js App Router for UI and Hono handlers mounted in `app/api/[...route]/route.ts` for API endpoints.
- Authentication is session-based using Prisma and PostgreSQL.
- Frontend data mutations and queries are managed with React Query.

Architecture

1. UI layer
- Route pages are thin wrappers.
- Feature cards contain form logic and user interactions.
- Shared primitives live in `components/ui`.

2. API layer
- Hono routes are grouped under `/api/auth`.
- Endpoints:
  - `POST /api/auth/login`
  - `POST /api/auth/register`
  - `POST /api/auth/onboarding`
  - `POST /api/auth/logout`
  - `GET /api/auth/me`

3. Data layer
- Prisma models include `User` and `Session`.
- Session rows store ownership and expiry.
- Cookie stores the session identifier used to resolve current user.

4. Validation strategy
- Zod schemas in `features/schemas.ts` are the source of truth for form payloads.
- Hono zValidator enforces request contracts server-side.
- React Hook Form + zodResolver enforces client-side validation.

Auth Lifecycle

1. Register
- Validates payload.
- Creates user row.
- Creates session row.
- Sets auth cookie.

2. Login
- Validates credentials.
- Creates session row.
- Sets auth cookie.

3. Resolve current user
- Reads cookie session id.
- Looks up session.
- Deletes expired session if expired.
- Loads user.

4. Logout
- Deletes session row by cookie id if present.
- Clears cookie.

Frontend Integration Pattern

1. Hooks
- One React Query hook per endpoint in `features/api`.
- Expected mutation behavior:
  - check `response.ok`
  - parse API error payload when not ok
  - throw `Error` for non-2xx
  - return JSON only on success

2. Components
- Submit actions call mutation hooks.
- Success path:
  - show success toast
  - navigate to next route
- Error path:
  - show user-friendly failure toast
- Buttons should use pending state to prevent duplicate submits.

Environment Notes

Required runtime configuration:
- `DATABASE_URL`
- `NEXT_PUBLIC_API_URL`
- `NODE_ENV`

Cookie guidance:
- Use `httpOnly: true`.
- Prefer `secure: process.env.NODE_ENV === "production"`.
- Prefer `sameSite: "lax"` for standard app navigation behavior.

Known Risks

1. Password storage
- Current flow compares plain text passwords.

2. Cookie policy
- `secure: true` in local non-https environments can block cookies.

3. Error throwing style
- Throwing plain objects in hooks can break error typing expectations.

Recommended Immediate Tasks

1. Security
- Add bcrypt or argon2 hashing for register/login.

2. Hook consistency
- Ensure all auth mutations throw `Error` on non-2xx.

3. UX guards
- Redirect unauthenticated users from protected routes.
- Redirect authenticated users away from sign-in/sign-up.

4. Quality
- Add integration tests for auth endpoints.
- Add an end-to-end happy path test for register -> onboarding -> dashboard -> logout.
